# JavaScript type coercion

JavaScript has built-in automatic type coercion, which some would call a feature and some would call a recipe for disaster.

It sure definitely can result in a lot of unexpected behaviour as it's not always very predictable from a human sense.

look at these examples

```javascript
'5' + 3; // "53"  (number → string)
'5' - 3; // 2     (string → number)
true + 1; // 2     (boolean → number)
null + 1; // 1     (null → 0)
undefined + 1; // NaN
```

As you can see datatypes get changed automatically based on which datatypes and operations that are being made.

You can also explicitly do this coerce

```javascript
Number('42'); // 42
String(42); // "42"
Boolean(0); // false
```

## Falsy values

All of the following values are falsy, which means that they will coerce to false.

```javascript
(false, 0, -0, '', null, undefined, NaN);
```

This means that you can perform checks on a variable and if they are assigned one of those values it will be false

```javascript
const object = undefined;

if (object) {
    //won't trigger
    doSomething();
}
```

## == vs ===

When compare data in JavaScript there's a big difference if you use == or ===

Using == will trigger type coercion which will allow some weird behaviours like comparing the value of a number with a boolean.

```javascript
0 == false; // true  ← coercion
'' == false; // true
```

Using === will not trigger coercion which means that it will not convert the datatype. For comparisons to be true both datatype and value must be the same.

```javascript
0 === false; // false ← strict, no coercion
null === undefined; // false
1 === 1; // true
```
